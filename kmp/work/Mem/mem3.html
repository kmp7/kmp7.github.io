Memory LLM

Метафора памяти в языковых моделях

В контексте больших языковых моделей термин «память» используется метафорически и охватывает несколько технологических уровней хранения и обработки информации. Понимание этих уровней критично для эффективной работы с LLM.

В отличие от человеческой памяти, «память» LLM — это каскад технических механизмов с разными сроками жизни, правилами доступа и функциональными целями.

Иерархия уровней памяти

Веса модели — это «долговременная память», сформированная в процессе обучения. Содержит обобщённые знания о языке, фактах, закономерностях. Не изменяется в процессе использования модели (inference).

Контекстное окно — «оперативная память» модели. Включает информацию из текущего диалога или обрабатываемого текста в пределах лимита токенов. Именно эту информацию модель активно учитывает при генерации каждого следующего токена.

KV-Cache — технический кэш промежуточных вычислений. Хранит ключи (Keys) и значения (Values) механизма внимания для уже обработанных токенов. Служит для ускорения генерации, избегая повторных вычислений.

Внешние механизмы — RAG (Retrieval-Augmented Generation), векторные базы данных, поисковые системы. Обеспечивают доступ к актуальной информации, которая не хранится внутри модели, но может быть включена в контекст.

Сессионная и глобальная память — реализуется на уровне сервиса (приложения), а не самой модели. Позволяет сохранять историю взаимодействий между сессиями.

Важно: Сама модель (нейронная сеть) не «помнит» предыдущие диалоги. Эффект памяти создаётся путём включения истории разговора в контекстное окно при каждом новом запросе.

Понятие контекстного окна

Почему контекст критичен для LLM

Языковые модели работают по принципу условной генерации: каждый следующий токен генерируется с учётом всех предшествующих токенов в контексте. Математически это выражается как:

P(токен_n | токен_1, токен_2, ..., токен_n-1)

Контекст определяет буквально всё: тему, стиль, язык, формат ответа, ограничения и задачу. Без контекста модель генерирует «усреднённый» текст, соответствующий распределению обучающих данных. С контекстом — текст, релевантный конкретной ситуации.

  • Идентичная модель может вести себя как поэт, программист, переводчик или аналитик — в зависимости от контекста
  • Контекст — это единственный способ «настроить» поведение модели без переобучения
  • Качество контекста напрямую определяет качество генерации

Техническая природа контекстного окна

Формат входных данных:

Модель работает не с текстом напрямую, а с токенами — числовыми идентификаторами, представляющими фрагменты текста (слова, части слов, символы). Входная последовательность включает:

  • Обычные токены — представления текста пользователя и модели
  • Специальные токены — маркеры начала/конца текста, разделители ролей (система/пользователь/ассистент), технические индикаторы
  • Позиционные кодировки — информация о положении токена в последовательности

Ограничение длины (токен-лимит):

Каждая модель имеет максимальную длину контекста, определяемую:

  • Архитектурно: размер позиционных кодировок, обученных во время тренировки
  • Вычислительно: механизм внимания (attention) имеет квадратичную сложность O(n²) по длине последовательности — удвоение длины увеличивает вычисления в 4 раза
  • По памяти: KV-cache растёт линейно с длиной контекста и количеством слоёв
Модель / Семейство Размер контекстного окна Примерно в словах*
GPT-3.5 4K — 16K токенов 3 000 — 12 000 слов
GPT-4 8K — 128K токенов 6 000 — 96 000 слов
Claude 3 200K токенов ~150 000 слов
Gemini 1.5 Pro до 1M токенов ~750 000 слов
Llama 3 8K — 128K токенов 6 000 — 96 000 слов

KV-Cache и механизм внимания

При генерации текста модель применяет механизм self-attention, который для каждого токена вычисляет три вектора:

  • Query (Q) — «запрос» текущего токена: что он ищет в контексте
  • Key (K) — «ключ» токена: как он может быть найден другими
  • Value (V) — «значение» токена: что он привносит, когда найден

При авторегрессивной генерации (токен за токеном) ключи и значения уже обработанных токенов не меняются. KV-Cache сохраняет эти вычисления, чтобы не пересчитывать их заново для каждого нового токена.

Практический эффект: генерация 100 токенов без KV-cache потребовала бы вычисления внимания для последовательностей длиной 1, 2, 3, ..., 100 токенов. С KV-cache каждый новый токен «смотрит» на кэшированные представления, что радикально ускоряет процесс.

Пояснение: KV-Cache — причина, по которой длинные контексты требуют много GPU-памяти. Для модели с 32 слоями и размерностью 4096 кэш для 100K токенов может занимать десятки гигабайт.

Роли контекстного окна в работе модели

Контекст как краткосрочная «память»

Контекстное окно выполняет функцию рабочей памяти для текущей задачи:

  • Хранение фактов: имена, даты, числа, упомянутые в диалоге
  • Отслеживание состояния: что уже обсуждено, какие решения приняты
  • Поддержание идентичности: характер персонажа, роль ассистента, стиль общения
  • Референция: разрешение местоимений и отсылок («он», «это», «как было сказано выше»)

Без этой информации в контексте модель не способна поддерживать связный многоходовый диалог или работать с документом как с единым целым.

Согласованность и тематическое единство

Контекст обеспечивает когерентность генерации на нескольких уровнях:

  • Тематическая согласованность: модель продолжает обсуждение в рамках заданной темы
  • Стилистическое единство: сохранение регистра, терминологии, тона
  • Логическая последовательность: следование установленным посылкам и ограничениям
  • Форматная консистентность: если начали список — продолжаем список, если код — код

Механизм внимания позволяет каждому генерируемому токену «видеть» весь предшествующий контекст и выбирать наиболее релевантную информацию для текущего момента.

Распределение внимания и приоритизация

Не все части контекста равнозначны для модели. Механизм внимания динамически распределяет «веса важности»:

  • Позиционные эффекты: токены в начале и конце контекста часто получают больше внимания (эффект «первичности» и «недавности»)
  • Семантическая релевантность: токены, семантически связанные с текущей генерацией, получают больший вес
  • Синтаксические связи: грамматически связанные элементы (подлежащее-сказуемое, определение-определяемое) сильнее связаны вниманием
  • Специальные токены: системные инструкции и маркеры ролей могут иметь особый статус
Важно: Распределение внимания — не статическое свойство, а результат взаимодействия текущего генерируемого токена со всем контекстом. Для разных выходных токенов распределение будет разным.

Измерение и оценка влияния контекста

Исследователи используют различные методы для изучения того, как контекст влияет на генерацию:

Количественные метрики:

  • Перплексия (Perplexity) — мера «удивлённости» модели. Более низкая перплексия = более предсказуемая генерация. Богатый релевантный контекст снижает перплексию.
  • Точность ответов — для задач с проверяемыми ответами (вопросно-ответные системы, извлечение фактов)
  • Согласованность диалога — автоматические метрики (BLEU, ROUGE для сравнения с эталонами) и человеческая оценка

Экспериментальные подходы:

  • Ablation studies: контролируемое удаление частей контекста и измерение изменения качества
  • Needle-in-a-haystack: размещение ключевого факта в разных позициях длинного контекста и проверка, находит ли его модель
  • Противоречивая информация: введение конфликтующих фактов для изучения приоритизации

Визуализация и интерпретация:

  • Attention maps: визуализация весов внимания между токенами
  • Attribution methods: оценка вклада каждого входного токена в конкретный выходной
  • Probing classifiers: анализ внутренних представлений модели
Метод оценки Что измеряет Ограничения
Перплексия Общую «уверенность» модели Не отражает фактическую правильность
Needle-in-a-haystack Способность находить факт в контексте Искусственная задача, может не отражать реальное использование
Attention visualization Паттерны «внимания» модели Внимание ≠ причинно-следственное влияние
Человеческая оценка Воспринимаемое качество Дорого, субъективно, плохо масштабируется

Работа с контекстным окном

Практические эффекты на генерацию

Эффект «затухания» (decay) ранних токенов:

Информация, расположенная в начале длинного контекста, может оказывать меньшее влияние на генерацию, чем информация в середине или конце. Этот феномен называют «lost in the middle» (потерянное в середине) — парадоксально, но середина длинного контекста часто наименее «видима» для модели.

Типичные проблемы при работе с длинным контекстом:

  • Повторение: модель может «забывать», что уже упоминала определённую информацию, и повторять её
  • Потеря фактов: конкретные детали из ранних частей контекста могут игнорироваться
  • Дрейф темы: постепенное отклонение от изначальной задачи
  • Артефакты обрезки: при автоматическом усечении контекста могут теряться критичные инструкции или контекст
  • Противоречия: модель может генерировать утверждения, противоречащие ранее сказанному в контексте
Пояснение: «Lost in the middle» — эмпирически наблюдаемый феномен. Если ключевой факт находится в начале или конце контекста, модель находит его лучше, чем если он в середине. Это важно учитывать при структурировании длинных документов.

Стратегии эффективного использования контекста

1. Сжатие и конденсация информации:

  • Создание кратких сводок (summaries) длинных документов или истории диалога
  • Выделение ключевых фактов и их явное перечисление
  • Удаление избыточной информации (приветствия, повторы, несущественные детали)
  • Использование структурированных форматов (списки, таблицы) вместо прозы

2. Sliding window (скользящее окно) и сегментация:

  • Разбиение длинного документа на перекрывающиеся сегменты
  • Обработка каждого сегмента отдельно с последующей агрегацией результатов
  • Сохранение «перекрытия» для поддержания контекста между сегментами

3. Внешнее хранение и RAG:

  • Хранение обширной информации во внешней базе (векторная БД)
  • Динамическое извлечение только релевантных фрагментов по запросу
  • Включение извлечённой информации в контекст перед генерацией

4. Prompt engineering — управление структурой контекста:

  • Размещение критичных инструкций в начале И в конце контекста
  • Явное структурирование: секции, заголовки, разделители
  • Разделение инструкций (как отвечать) и данных (на основе чего)
  • Использование примеров (few-shot) для демонстрации желаемого формата
Стратегия Когда применять Потенциальные риски
Сжатие/конденсация Длинные диалоги, многостраничные документы Потеря нюансов, ошибки при автоматическом сжатии
Sliding window Очень длинные тексты, книги, кодовые базы Потеря глобального контекста, сложность агрегации
RAG Обширные базы знаний, актуальные данные Ошибки поиска, нерелевантные извлечения
Структурированные промпты Всегда полезно Увеличение длины контекста за счёт структуры

Контекст vs. инструкции: баланс

Контекстное окно должно вмещать несколько типов информации, конкурирующих за ограниченное пространство:

  • Системные инструкции: роль, ограничения, формат ответа
  • Примеры (few-shot): демонстрация желаемого поведения
  • Фоновые данные: документы, база знаний, история
  • Текущий запрос: непосредственный вопрос или задача

Принципы балансировки:

  • Инструкции должны быть достаточно детальными, но не избыточными
  • Примеры эффективнее абстрактных описаний, но занимают больше места
  • Не вся доступная информация должна быть в контексте — только релевантная
  • При дефиците места — приоритет текущей задаче перед историей
Важно: Токены, потраченные на нерелевантную информацию, — это не просто «потраченное место». Они могут активно ухудшать качество генерации, отвлекая внимание модели и создавая шум.

Перспективы контекстных механизмов

Архитектурные подходы к расширению контекста

Длинно-контекстные трансформеры:

Исследователи разрабатывают архитектуры, способные эффективно работать с длинными последовательностями:

  • Sparse Attention: вместо полного внимания (каждый токен к каждому) — разреженные паттерны (локальное + глобальное внимание). Примеры: Longformer, BigBird.
  • Linear Attention: замена квадратичного softmax-внимания на линейные приближения. Снижает сложность до O(n).
  • Компрессирующие слои: специальные механизмы для «сжатия» ранних частей контекста в более компактные представления.
  • Рекурсивные/иерархические механизмы: обработка текста на нескольких уровнях абстракции.

Позиционные кодировки для длинных последовательностей:

  • ALiBi (Attention with Linear Biases): позиционная информация через bias в attention, легко экстраполируется на длины больше обучающих
  • RoPE (Rotary Position Embedding): ротационные кодировки, хорошо масштабирующиеся
  • NTK-aware scaling: методы адаптации позиционных кодировок для длин вне распределения обучения

Вычислительные ограничения и решения

Основные ограничения:

  • Память GPU: KV-cache для длинных контекстов требует гигабайты памяти
  • Время вычислений: даже с оптимизациями, длинный контекст = медленнее генерация
  • Энергопотребление: обработка длинных контекстов требует больше вычислительных ресурсов

Технические решения:

  • FlashAttention: оптимизированная реализация внимания, снижающая требования к памяти
  • Quantization: снижение точности KV-cache (16-bit → 8-bit → 4-bit)
  • Paged Attention: динамическое управление памятью для KV-cache (как в vLLM)
  • KV-cache compression: сжатие или отбрасывание менее важных ключей/значений
Подход Преимущество Компромисс
Sparse Attention Линейная/субквадратичная сложность Потеря некоторых дальних связей
KV-cache quantization 2-4x снижение памяти Потенциальная деградация качества
Компрессия контекста Больше информации в меньшем пространстве Потеря деталей при сжатии
Гибридные RAG-системы Практически неограниченные знания Зависимость от качества поиска

Перспективы развития

Тренды для пользователей:

  • Увеличение контекстных окон: от 4K → 128K → 1M+ токенов за несколько лет
  • Гибридные подходы: комбинация встроенного контекста с внешним поиском
  • Автоматическое управление контекстом: системы, автоматически оптимизирующие, что включать в контекст
  • Персистентная «память»: механизмы долгосрочного хранения информации о пользователе/задаче

Вероятные направления исследований:

  • Обучение моделей явно работать с очень длинными зависимостями
  • Развитие компрессирующих механизмов, сохраняющих семантику
  • Интеграция явной символической памяти с нейросетевыми представлениями
  • Архитектуры с динамически выделяемым контекстом в зависимости от задачи
Примечание: Несмотря на рост размеров контекстных окон, «эффективная» длина контекста (та, которую модель реально использует качественно) часто меньше максимальной. Bigger ≠ better во всех случаях.

Научный фронтир: открытые вопросы

Масштабирование и эффективность

Как эффективно масштабировать полезную длину контекста при ограниченных ресурсах?

Увеличение максимальной длины контекста технически возможно, но это не гарантирует, что модель будет эффективно использовать всю эту длину. Центральный вопрос: как добиться того, чтобы модель реально «видела» и использовала информацию из всех частей длинного контекста, а не только из начала и конца?

Какие архитектурные изменения дают лучшую «полезную» длину контекста?

  • Линейные/субквадратичные механизмы внимания vs. полное внимание
  • Компрессирующие слои и их влияние на сохранение информации
  • Рекурсивные и многоуровневые архитектуры
  • Гибридные подходы: локальное + глобальное внимание

Пока нет консенсуса, какой подход оптимален для разных типов задач.

Сжатие и сохранение информации

Как оптимально сжимать ранние части контекста?

При работе с длинными последовательностями неизбежно встаёт вопрос о сжатии. Но как сжимать так, чтобы сохранить семантическую и прагматическую релевантность для будущих предсказаний? Какая информация критична, а какая может быть отброшена?

Task-aware compression:

Как формализовать «полезность» токена для сохранения в зависимости от задачи? Токен, важный для суммаризации, может быть неважен для ответа на вопрос, и наоборот. Существующие методы компрессии часто task-agnostic, что ограничивает их эффективность.

Затухание влияния и «забывание»

Механизмы и закономерности «затухания»:

Эмпирически наблюдается, что влияние ранних токенов ослабевает. Но насколько это предсказуемо? Зависит ли от типа информации (факты vs. инструкции vs. стиль)? Можно ли компенсировать это специальным обучением или архитектурными решениями?

Почему модель «забывает» релевантные факты?

Иногда модель игнорирует явно представленную в контексте информацию, даже когда она кратковременно релевантна. Механизмы этого феномена до конца не поняты. Это может быть связано с:

  • Конкуренцией за «внимание» между токенами
  • Влиянием обучающих данных (более частые паттерны доминируют)
  • Позиционными эффектами
  • Форматом представления информации

Атрибуция и интерпретируемость

Как измерять «вклад» контекстного фрагмента в предсказание?

Существующие методы атрибуции (attention weights, gradient-based methods, perturbation studies) имеют ограничения. Веса внимания не обязательно отражают причинно-следственное влияние. Пертурбационные методы (удаление/изменение частей контекста) дороги вычислительно и могут давать артефакты.

Разделение типов информации в контексте:

Как модель внутренне различает инструкции, метаданные и факты? Как это влияет на баланс между следованием инструкциям и информационной точностью? Когда инструкция конфликтует с фактами в контексте, что побеждает и почему?

Разрешение противоречий

Как модель обрабатывает противоречивую информацию?

Когда в контексте присутствует конфликтующая информация, модель должна как-то разрешить противоречие. Но механизмы этого разрешения непрозрачны:

  • Какая информация получает приоритет? (Более поздняя? Более частая? Более явная?)
  • Можно ли моделировать «доверие» к разным источникам внутри контекста?
  • Как влияет позиция конфликтующих фактов в контексте?

Влияние обучения

Как свойства обучающего корпуса влияют на использование контекста?

Длина последовательностей в обучении, структура диалогов, распределение типов текстов — всё это формирует «привычки» модели работать с контекстом. Но связь между характеристиками обучающих данных и поведением модели в inference недостаточно изучена.

Перенос навыков работы с длинным контекстом:

Если модель научилась эффективно работать с длинным контекстом на одном типе задач/доменов, переносится ли этот навык на другие? Или для каждого домена нужна специфическая адаптация?

Пояснение: Этот раздел представляет активные направления исследований. Многие вопросы не имеют окончательных ответов, и именно они определяют фронтир развития технологии.

Страна советов

Формирование запросов для лингвистического анализа

При использовании LLM для лингвистических задач критично понимать, как структура и содержание контекста влияют на качество результатов.

Общие принципы:

  • Явно формулируйте задачу: не «проанализируй текст», а «выдели именные группы с определениями-причастными оборотами»
  • Указывайте формат выхода: список, таблица, аннотированный текст, JSON
  • Приводите примеры: 2-3 примера разметки или анализа в желаемом формате эффективнее длинных объяснений
  • Определяйте терминологию: если используете специальные термины, убедитесь, что модель понимает их так же, как вы

Лучшие практики для типичных задач

Морфологический и синтаксический анализ:

  • Работайте с небольшими порциями текста (1-5 предложений) для детального анализа
  • Явно указывайте систему разметки (UD, ваша собственная, другая)
  • Для нестандартных тегов или категорий обязательно приводите примеры

Семантический анализ:

  • Разделяйте уровни: лексическая семантика, пропозициональная, дискурсивная
  • При работе с многозначностью указывайте, нужно ли разрешение или перечисление значений
  • Для анализа метафор/метонимии предоставляйте достаточный контекст употребления

Корпусные исследования:

  • При работе с большими объёмами — разбивайте на части, агрегируйте результаты
  • Для подсчётов и статистики используйте структурированные форматы вывода
  • Критически проверяйте результаты на выборке — LLM могут «галлюцинировать» примеры

Типичные ошибки и как их избежать

Ошибка 1: Перегрузка контекста нерелевантной информацией

Включение всего текста исследования, когда нужен анализ одного абзаца. Это не только тратит токены, но может ухудшить качество за счёт «шума».

Решение: Включайте только релевантный фрагмент + минимальный необходимый контекст.

Ошибка 2: Неявные допущения

Предположение, что модель «понимает» вашу теоретическую рамку или терминологию без объяснений.

Решение: Явно определяйте термины и теоретические понятия, особенно если они используются нестандартно.

Ошибка 3: Слишком длинные диалоги без суммаризации

Накопление десятков сообщений в истории диалога, пока контекст не переполнится.

Решение: Периодически создавайте сводку ключевых договорённостей и результатов, начинайте новую «сессию» со сводкой.

Ошибка 4: Игнорирование позиционных эффектов

Размещение критичных инструкций в середине длинного промпта.

Решение: Ключевые инструкции — в начало и/или конец. Для сложных задач — повторяйте важные ограничения.

Ошибка 5: Отсутствие примеров

Абстрактное описание задачи без конкретных примеров желаемого результата.

Решение: 2-3 качественных примера (few-shot) почти всегда улучшают результат.

Задача Рекомендуемая структура промпта
Морфологическая разметка [Инструкция + система тегов] → [2-3 примера с разметкой] → [Текст для разметки] → [Формат вывода]
Извлечение конструкций [Определение конструкции + примеры] → [Текст для поиска] → [Формат списка результатов]
Анализ дискурса [Теоретическая рамка] → [Параметры анализа] → [Текст] → [Структура отчёта]
Сравнительный анализ [Критерии сравнения] → [Текст A] → [Текст B] → [Формат сравнительной таблицы]
Важно: LLM — инструмент, а не экспертная система. Всегда критически оценивайте результаты, особенно для исследовательских целей. Модель может уверенно генерировать неверный анализ. Используйте LLM для ускорения работы и генерации гипотез, но верифицируйте результаты.
Примечание: Разные модели могут по-разному справляться с лингвистическими задачами. Если одна модель даёт неудовлетворительные результаты, попробуйте другую или переформулируйте запрос. Эксперименты с формулировками — нормальная часть работы с LLM.
